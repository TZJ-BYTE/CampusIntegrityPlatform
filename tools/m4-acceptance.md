# M4 验收用例 D（两设备离线 → 联机自动收敛）

目标：验证“离线可用 + 联机后自动同步（云存储体验）”，并覆盖常见失败场景（断网/重启/重复 push/pull/大量积压）。

## 1. 准备

- 启动服务端（Go 一体化服务，默认端口 8788）：

```powershell
cd d:\project\rustproject\CampusIntegrityPlatform
npm run server
```

- 准备两份“便携版数据目录”（模拟两台设备，互不影响）：

```powershell
$root = "d:\project\rustproject\CampusIntegrityPlatform"
$a = Join-Path $root "sandbox\device-a"
$b = Join-Path $root "sandbox\device-b"
New-Item -ItemType Directory -Force -Path (Join-Path $a "data") | Out-Null
New-Item -ItemType Directory -Force -Path (Join-Path $b "data") | Out-Null
```

说明：本应用在“便携模式”下会把 `user.db/content.db` 写入 `app.exe` 同目录的 `data/`。因此只要把可执行文件分别放到 `device-a/` 与 `device-b/`，就会得到两套独立数据。

## 2. 操作步骤

### 2.1 设备 A（离线操作）

1) 断网（或不登录）
2) 操作：
- 收藏 2 条内容（案例/故事均可）
- 完成 1 次答题（产生积分流水）
- 修改 1 项设置（任意）
3) 打开头像 → 账号与云同步：确认“待同步条数”增加

### 2.2 设备 B（离线操作）

1) 同样断网（或不登录）
2) 操作：
- 收藏另外 2 条内容（不要与 A 完全相同）
- 完成 1 次答题
3) 确认“待同步条数”增加

### 2.3 联机并登录（两边都登录到同一账号）

1) 恢复网络
2) 设备 A：头像 → 账号与云同步 → 登录（用户名固定，例如 `test`）
3) 等待自动同步触发（或切到别的页再回来），观察：
- 云状态从“未连接”变“同步中/已连接”
- 待同步条数逐步归零
4) 设备 B：用同一账号登录，等待自动同步完成

## 3. 验收点（结果一致性）

- 收藏合并：A 与 B 最终“我的收藏”一致（并集）
- 积分合并：积分不会翻倍；重复同步不会重复加分
- 设置策略：设置以最后写入为准（若两端同一字段冲突）
- 幂等：重复打开/关闭应用、多次触发同步，不会产生重复收藏/重复积分

## 4. 失败场景清单（做不到也要记录）

- 断网时：离线操作不受影响；仅同步失败（诊断可见），不弹打断式错误
- 服务端重启：重启后不丢 token/users/events；客户端可继续同步
- 大量积压：离线产生很多事件后，单次 `sync_run` 能分批完成 push/pull（避免需要用户手动点多次）
- 重复 push/pull：同 event_id 重复提交不会导致数据翻倍

## 5. 已知限制（现阶段可接受）

- “两设备”模拟需要两份可执行文件（便携模式）；若操作系统限制同名应用多开，需要把两个可执行文件命名为不同文件名。
- 用户信息（昵称/头像主色）已进入同步域；头像图片数据目前仅本机保存（不参与同步）。
