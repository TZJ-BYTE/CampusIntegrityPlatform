# 内容库后台（CMS）可行性与方案草案

目标：让你不用手改服务器文件/数据库，也能通过网页把“法规/故事/场所/案例”等公共内容增删改，并一键发布到客户端的内容更新链路（versions.json + content-pack.zip）。

## 1. 现状与可行性结论

当前项目已具备“发布/分发/客户端更新”的骨架：

- 服务端（Go）以静态文件方式提供：
  - `GET /versions.json`
  - `GET /content-pack.zip`
- 客户端会定时检查 `versions.json`，发现 `latest.version` 与本地 `content.db` 版本不同后，下载并应用 `content-pack.zip`（替换本地内容库）。
- 构建侧已有 `build_content_pack`（Rust bin）能生成 `content.db + manifest.json + versions.json` 并打包 zip。

因此：做一个 CMS 只要能产出“可发布的 content-pack.zip + versions.json”，就能让客户端同步你对内容的增删改，完全可行。

## 2. 内容库 vs 同步库（再次澄清）

- 内容库（content.db）：公共内容；通过“版本包”整体更新；你在 CMS 里做的增删改属于这类。
- 同步库（user.db + 事件）：个人数据（收藏/积分/答题/用户信息）；走增量同步；不建议用 CMS 去直接改。

本 CMS 仅覆盖内容库。

## 3. CMS 的两种落地形态（推荐路线）

### 方案 A：独立 Web 后台（推荐）

一个独立的管理站点（浏览器打开），提供：

- 内容管理（CRUD）：法规/故事/场所/案例/题库
- 媒体管理：图片/视频封面上传、资源库、引用关系
- 发布中心：版本号、发布说明、生成包、回滚
- 权限与审计：登录、角色、操作日志

发布后把 `versions.json/content-pack.zip` 写入现有 `server-go/content-repo/`（或对象存储），客户端照常更新。

优点：专业、可多端协作、适合真数据进入生产。
缺点：要做登录/权限/安全与发布流程的收口。

### 方案 B：客户端内置“内容编辑模式”（备选）

在现有客户端里加一个“管理员入口”，对本地内容库编辑，然后“导出发布包”。

优点：UI 最像客户端、复用组件多。
缺点：权限/发布控制不如独立后台清晰，多人协作也不方便。

接下来默认按方案 A 推进（你已确认）。

## 3.1 访问控制（你的最新要求）

CMS 不做用户体系/角色，但需要“密码锁”，部署到服务器后可被任何人访问到登录页/密码输入页，未通过验证不能进入编辑与发布页面。

推荐采用两层保护（可二选一，但两层最稳）：

1) 反向代理层（推荐）：Nginx/Caddy 做 HTTP Basic Auth
- 实现最简单、最可靠、不会把 CMS 自身暴露在未授权请求下
- 配合 HTTPS 使用

2) 应用层（补强）：CMS 自己做单密码会话
- 密码不写进仓库；用环境变量提供（例如 `CMS_PASSWORD_HASH`）
- 使用安全哈希（bcrypt/argon2）存储与校验
- 登录后发会话 cookie（HttpOnly + Secure + SameSite），做到“只输入一次密码”
- 给写操作加 CSRF 防护与基本的限流/失败次数限制

发布接口与删除/批量操作必须在鉴权之后才可访问。

### 一次输入密码的会话策略（推荐）

- 登录成功后设置 `cms_session` cookie（HttpOnly + Secure + SameSite=Lax）
- 会话有效期：默认 12 小时（可配置）
- 可选“记住我”：有效期 7~30 天（仍建议仅用于 HTTPS 部署）
- 提供“退出登录”：清理 cookie
- 防暴力：失败次数限制与短暂冷却（例如 10 次失败锁 5 分钟）

## 4. 页面信息架构（UI 结构）

风格：与客户端尽量一致（同配色、同卡片/信息流），但交互为“编辑优先”。整体布局建议为“三栏”：

1) 左侧：模块与筛选
- 内容类型：法规 / 故事 / 场所 / 案例 / 题库
- 状态筛选：草稿 / 已发布 / 已归档
- 标签/层级/来源筛选（按内容类型变化）

2) 中间：列表（可表格/卡片切换）
- 支持搜索、排序、分页/无限滚动
- 支持批量选择：批量删除/批量改标签/批量发布
- 列表项展示缩略图（封面）、标题、来源、更新时间、状态

3) 右侧：编辑器 + 预览（可切 Tab）
- Tab A：编辑（表单/富文本/条款编辑）
- Tab B：预览（尽量模拟客户端详情页展示）
- Tab C：历史/版本（草稿历史、发布版本、差异对比）

顶栏（固定）：全局搜索、草稿提醒、发布入口、账号与权限入口。

## 5. 各内容类型的编辑形态（最小可用 → 可扩展）

### 法规（regulations）
- 基本信息：标题、层级、来源、发布日期、关键词
- 条款（sections）：可视化条款编辑器（支持拖拽排序、拆分、合并）
- 学习资源（media）：封面图、导学视频、配图/附件

### 场所（venues）
- 基本信息：名称、类型、地点、开放时间、联系方式
- 介绍：富文本
- 参观资源：封面图/导览视频/相册

### 案例（cases）
- 标题、场景、摘要
- 情景描述、违纪风险、正确做法
- 配图/视频（可选）

### 故事（stories）
- 标题、来源、正文、日期/排序字段
- 封面（可选）、配图（可选）

## 6. “图片/视频”如何设计才不堵后路（重要）

你希望未来能加视频/照片，同时客户端有离线诉求。建议分两阶段：

### 阶段 1（最小可用，先让编辑起来）
- CMS 支持上传媒体，但发布包里只写“媒体元信息”（封面/视频 URL 等）。
- 客户端优先显示占位图，若联网则加载远程资源（或只加载封面图）。

优点：实现快，发布包不变；缺点：离线时媒体不可用。

### 阶段 2（离线媒体，长期形态）
- content-pack.zip 增加 `assets/` 目录（图片/视频/封面），`content.db` 只存相对路径与元信息。
- 客户端 `content_apply_pack` 解压时同时落盘 assets，并提供读取路径给前端。

优点：离线完整；缺点：需要扩展打包与客户端解包逻辑（属于后续开发项）。

当前项目已先在前端做了媒体占位组件与封面占位素材，能平滑承接阶段 1 → 阶段 2。

## 7. 发布工作流（CMS → 客户端可见）

目标：你在 CMS 里删假数据、加真数据后，客户端能自动更新到同样的结果。

推荐发布流：

1) 编辑内容（草稿）
2) 校验（必填字段、ID 唯一、引用关系、条款结构）
3) 生成版本：
   - 生成 content.db
   - 写入 `meta.content_version = <version>`
   - 打包 content-pack.zip（manifest.json + content.db；后续可加 assets）
4) 发布：
   - 更新 `versions.json.latest.version/url/notes`
   - 将产物写到 `server-go/content-repo/`（或对象存储）
5) 客户端自动更新：
   - 下次检查发现版本变化 → 下载并应用

## 8. 版本与回滚策略（建议）

- 版本号：建议语义化或递增数字（至少保证每次发布变更 version）
- notes：写“本次更新摘要”（用于客户端展示）
- 回滚：保留最近 N 个发布包与 `versions.json` 历史（发布中心一键回退）

## 9. 与现有代码的最小集成点（不改客户端也能跑）

只要 CMS 能把产物写成下面两个文件：

- `${CONTENT_REPO_DIR}/versions.json`
- `${CONTENT_REPO_DIR}/content-pack.zip`

现有 server-go 直接静态返回，客户端就能更新。

构建层可复用现有 `build_content_pack` 的打包格式与 manifest 结构，CMS 只需替换“写入 content.db 的数据来源”即可。
